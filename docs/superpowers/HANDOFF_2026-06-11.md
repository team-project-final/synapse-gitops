## 2. bring-up이 EKS에서 ES를 못 띄우던 결함 4건 (PR #180) — 라이브 디버깅

ES가 bring-up에 없어 **EKS에서 한 번도 제대로 기동된 적 없던** 결함이 연쇄로 드러남:
1. **fsGroup 누락**: EBS 볼륨 root 소유→ES(uid1000) node.lock 못써 `failed to obtain node locks` CrashLoop. `securityContext.fsGroup:1000`. **함정: fsGroup chown은 새 pod 마운트 시에만** → spec 추가해도 기존 CrashLoop pod는 `delete pod`로 재생성해야 적용(다음 윈도우는 처음부터 fsGroup이라 자동).
2. **ES/schema-registry 미배포**: ApplicationSet 목록 밖 standalone App(`argocd/elasticsearch.yaml`·`schema-registry.yaml`)인데 phase_manifests 미적용. 추가.
3. **gp3 StorageClass 순서**: observability(후순위)→manifests 선두 이동(ES PVC가 더 일찍 필요).
4. **es-reindex 레이스**: ArgoCD reconcile 비동기 대비 StatefulSet 생성 대기+graceful skip.

**환경 함정(PR #181)**: Git Bash+Windows kubectl.exe가 MSYS 경로(`/tmp`) KUBECONFIG 무시→localhost:8080 폴백("API 도달 실패"). `cygpath -m` 변환. + stale tf lock(force-unlock)·갓생성 클러스터 첫 터널 폴링 부족도 재현.
