## 2. 결정 사항 (브레인스토밍 합의)

| 결정 | 선택 | 근거 |
|------|------|------|
| #121 도메인 | nip.io/sslip.io 임시 도메인 | 실 도메인 부재, 폐기 전제 데모 검증에 충분 |
| #121 TLS 종료 | **ALB + self-signed ACM import** | 리포가 ALB(`aws-load-balancer-controller`)·gateway·prod netpol에 전적으로 투자됨. cert-manager는 ALB HTTPS 리스너를 구동 못함(ALB는 ACM 인증서로 종료). 일관성 최상 |
| window 구조 | 단일 통합 window (접근 A) | 클러스터를 한 번만 띄워 비용·완결성 모두 유리 |
| 코드 vs 검증 분리 | Phase 0 사전 머지(무비용) + window 내 apply·치환·검증 | 과금 시간 최소화 |

> self-signed의 "브라우저 경고" 성격은 ALB-import든 cert-manager든 동일하다. 차이는 오직 **TLS 종료 지점(ALB vs nginx)** 이며, 리포 일관성을 위해 ALB를 택했다.

## 3. Window 구조와 실행 순서

#121의 self-signed 인증서 SAN은 **ALB IP가 정해져야** 생성 가능(런타임 의존). 따라서 "사전 작성(무비용)"과 "window 내 런타임 치환"을 분리한다.

```
Phase 0 — 사전 작성 (무비용, window 전 main 머지)
  ├─ #121 ingress 매니페스트(nip.io 호스트 + ALB self-signed 전제, ARN/IP는 플레이스홀더)
  ├─ scripts/gen-nipio-selfsigned.sh (ALB IP → CA+cert SAN 생성 → ACM import → ARN 반환)
  ├─ #122 Image Updater annotation 사전 점검 (이미 있으면 noop)
  └─ #120 kafka-topics terraform plan 검토 + .terraform.lock.hcl 커밋

Phase 1 — apply + 부트스트랩 (window 시작, 과금 ON)
  ├─ terraform apply (EKS) → bring-up.sh 부트스트랩
  │     (ArgoCD HA + ESO + ApplicationSet, dev auto-sync · staging/prod manual)
  └─ bastion에서 kafka-topics terraform apply (MSK TLS provider)

Phase 2 — 핵심 검증 (bastion SSM, 상호 독립 → 병렬 가능)
  ├─ #91/#92  verify-argocd-deploy.sh synapse-dev 5/5 + synapse-staging 5/5 (+ 롤백 1회 <3분)
  ├─ #120     토픽 목록 · ACL · openssl s_client TLS · console produce/consume
  └─ #122     이미지 푸시 → dev 자동 반영 E2E + 평균 반영시간(≤5분) + 잘못된 태그 롤백

Phase 3 — #121 외부 노출 검증 (ALB 의존 → Phase 1 이후)
  └─ ingress apply → ALB IP 획득 → gen-nipio 스크립트 → ACM ARN 치환 → 재apply
     → argocd UI nip.io HTTPS 접속 + CA 신뢰체인 유효 + dev 앱/webhook 외부 도달

Phase 4 — 정리
  ├─ 5 이슈 DoD 체크 + 각 결과 synapse-shared#20 통보 코멘트
  └─ terraform destroy (과금 차단)
```

**설계 포인트**
- Phase 0를 사전 머지로 분리 → 과금 시간 최소화(window에선 apply·치환·검증만).
- Phase 2는 서로 독립 → bastion 세션에서 병렬 진행해 window 시간 단축.
- #121만 ALB(Phase 1 산출물)에 의존 → Phase 3으로 분리.

## 4. #121 신규 작성 상세 (유일한 코드 작업)

### 산출물 (Phase 0, main 머지)

**A. `infra/ingress/nipio/argocd-server-ingress.yaml`** *(신규)*
현재 ArgoCD UI 외부 노출 ingress가 리포에 없음.
- ALB internet-facing → `argocd-server` 백엔드
- host `argocd.<ALB_IP>.nip.io` — **UI + webhook 엔드포인트(`/api/webhook`) 동시 충족**
- `certificate-arn: <ACM_ARN>` 플레이스홀더

**B. `infra/ingress/nipio/dev-ingress.yaml`** *(기존 dev gateway 패턴의 nip.io 변형)*
- host `dev.<ALB_IP>.nip.io` → `gateway:80` (리포의 단일 진입 gateway 패턴 유지)
- DoD "dev-\<app\>" 세분 패턴은 gateway 경로 라우팅(`/api/<app>`)으로 충족 — 별도 per-svc ingress를 신설하지 않아 ALB/netpol 일관성 유지
- 기존 `infra/ingress/dev-ingress.yaml`·`staging-ingress.yaml`(실 도메인+ACM용)은 **보존**(prod 사이클 자산)

**C. `scripts/gen-nipio-selfsigned.sh`** *(신규)*
- 입력: ALB DNS → `dig +short`로 IP 추출
- self-signed **CA + leaf 인증서** 생성, SAN = `argocd.<IP>.nip.io`, `dev.<IP>.nip.io`, `*.<IP>.nip.io`
- `aws acm import-certificate` → **ARN 출력** + CA(`ca.crt`)를 신뢰 검증용으로 emit

### 닭-달걀 해소 (Phase 3 런타임 순서)

```
1. argocd/dev ingress apply (cert-arn 미설정 → ALB 프로비저닝만 트리거)
2. ALB DNS 확보 → dig → 현재 IP
3. gen-nipio-selfsigned.sh <ALB_DNS> → cert SAN=해당 IP nip.io → ACM import → ARN
4. ingress에 certificate-arn=ARN + host nip.io 치환 → 재apply
5. curl --cacert ca.crt https://argocd.<IP>.nip.io → 200 + 체인 유효
```

> nip.io의 `<IP>`는 window 동안만 유효(ALB IP는 ephemeral) — 폐기 전제 검증이라 의도된 동작. "TLS 인증서 유효"는 **CA 신뢰 주입(`--cacert`) 기반 체인 검증**으로 충족(브라우저 단독 접속 시 경고는 self-signed 본질).

## 6. 리스크·롤백·통보

### 리스크와 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| ALB IP가 window 중 회전 | nip.io 호스트 무효화 | window 1회성 검증 — 발생 시 IP 재추출 + cert 재발급(스크립트 재실행). 폐기 전제라 수용 |
| ACM import 권한/리전 불일치 | #121 차단 | Phase 0에서 IAM 권한·리전 사전 점검 (cert는 ALB와 동일 리전) |

### 롤백 안전망

- 모든 인프라는 `terraform destroy`로 원복(검증 후 즉시).
- #91 롤백 테스트(`rollout undo`)·#122 롤백 테스트는 **검증 항목 자체** — 실패해도 클러스터는 destroy로 정리.
