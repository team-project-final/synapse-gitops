Synapse 로컬 MSA 개발환경 세팅 가이드

처음부터 따라 하면 내 PC에서 Synapse가 돌아갑니다 · 초보 ~ 준시니어용

0개요

이 가이드를 끝내면 내 PC에서 Synapse MSA 전체(또는 내가 맡은 서비스)가 돌아갑니다. 초보는 번호 단계만 위→아래로 따라가면 되고, 준시니어는 각 단계의 “▸ 심화” 박스를 펼쳐 원리와 트러블슈팅을 볼 수 있습니다. AWS/EKS 배포는 이 가이드의 범위가 아닙니다(→ §9 더 깊이 보기).

전체 구성도

아래 구성도의 노드를 클릭하면 포트·헬스·확인 명령이 오른쪽(모바일은 아래)에 표시됩니다.

노드를 클릭하면 포트·헬스·확인 명령이 표시됩니다.

구성 요약: Client → Gateway(경로② 전용) → 서비스 5개(platform 8080, engagement 8082, knowledge 8083, learning-card 8084, learning-ai 8000/8090) → Kafka 9092, Schema Registry 8085/8086, PostgreSQL 5432, Redis 6379, 검색엔진 9200.

서비스 간 비동기 통신은 Kafka + Avro(Schema Registry) 이벤트로 이뤄집니다. 서비스는 도메인별로 분리된 독립 Spring Boot(JDK 21) 앱이며, AI 서비스(learning-ai)만 Python/FastAPI입니다.

실행 경로

권장 메인 경로는 §3 로컬 k8s(minikube)입니다 — 인프라·Gateway·서비스를 모두 파드로 띄워 실 배포(EKS)와 가장 가깝습니다. 아래 두 가지는 docker compose 기반 대안(§4/§5)이며 서로 다른 인프라 스택을 씁니다.

왜 스택이 둘로 갈렸나? (심화)
두 compose는 검색엔진(Elasticsearch ↔ OpenSearch), Kafka(7.6.1 ↔ 7.7.0), Postgres(pgvector ↔ alpine), Schema Registry 호스트 포트(8085 ↔ 8086)가 다릅니다. synapse-gitops compose는 “제품 전체를 컨테이너로 띄우는(빌드 또는 ghcr 이미지) 배포 지향” 구성이고, synapse-shared compose는 “로컬 개발용 인프라 + Gateway, 앱은 본인이 소스로 실행”하는 개발 지향 구성입니다. 이 분기는 레포가 각자 진화하며 생긴 알려진 기술부채이며, 본 문서는 현황을 반영만 합니다(통합은 별도 과제).

AWS CLI·kubectl이 왜 없어도 되나? (심화)
로컬 환경은 모든 것을 Docker 컨테이너 + 소스 실행으로 구성합니다. AWS CLI와 kubectl은 EKS(클라우드) 클러스터에 접근할 때 필요한 도구이고, 로컬에는 클러스터가 없습니다. EKS·ArgoCD·Bastion 접근은 개발자 가이드와 런북이 담당합니다.

2레포 클론

모든 레포를 같은 부모 폴더 아래 형제(sibling)로 클론해야 합니다. compose가 ../synapse-* 상대경로로 다른 레포의 소스를 빌드하기 때문입니다.

cd C:\workspace\team-project-final git clone https://github.com/team-project-final/synapse-gitops.git git clone https://github.com/team-project-final/synapse-shared.git git clone https://github.com/team-project-final/synapse-gateway.git git clone https://github.com/team-project-final/synapse-platform-svc.git git clone https://github.com/team-project-final/synapse-engagement-svc.git git clone https://github.com/team-project-final/synapse-knowledge-svc.git git clone https://github.com/team-project-final/synapse-learning-svc.git

(부모 폴더 경로는 자유입니다. 위 예시는 C:\workspace\team-project-final를 씁니다.)

클론 후 폴더 구조

경로별로 실제 필요한 레포는? (심화)
경로① 풀 컨테이너는 synapse-gitops + 빌드 대상 서비스 레포 (platform/engagement/knowledge-svc, learning-svc)가 필요합니다(Gateway 없음). 경로② 하이브리드는 synapse-shared + synapse-gateway + 본인이 개발할 서비스 레포가 필요합니다. 처음엔 위 7개를 모두 받아두면 어느 경로든 바로 시작할 수 있습니다.

3로컬 k8s(minikube) — 메인 경로

권장 메인 경로. minikube 로컬 쿠버네티스에 인프라 + Gateway + 5개 서비스를 모두 파드로 띄웁니다. 실 배포(EKS)와 가장 가까운 형태입니다. local-k8s/ kustomize가 EKS용 매니페스트를 로컬용으로 적응합니다(ExternalSecret 제거 · 인프라 인클러스터 · 로컬 이미지). docker compose 경로는 §4/§5의 대안입니다.

minikube version kubectl version --client docker --version

형제 레포(§2의 7개)가 클론돼 있어야 합니다. 스크립트가 Gateway·서비스 이미지를 빌드해 minikube에 적재합니다(learning-card는 Dockerfile이 없어 bootBuildImage로 빌드 — 최초 다소 느림).

cd C:\workspace\team-project-final\synapse-gitops bash scripts/minikube-up.sh
kubectl -n synapse-local get pods
kubectl -n synapse-local port-forward svc/gateway 8080:80 kubectl -n synapse-local port-forward svc/learning-ai 8000:80

그 다음 Gateway 경유로 curl http://localhost:8080/api/platform/actuator/health, 브라우저로 http://localhost:8000/docs. 호출·확인 방법은 §6 참고(이 경로는 Gateway 포함 → /api/{서비스}/** 라우팅 동작).

3.5 클러스터 들여다보기 (뷰)

파드·서비스 상태와 로그를 보는 기본 명령입니다. 모든 리소스는 synapse-local 네임스페이스에 있습니다.

kubectl -n synapse-local get all # 파드/서비스/디플로이 한눈에 kubectl -n synapse-local get pods -o wide # 노드/IP 포함 kubectl -n synapse-local get pods -w # 상태 실시간 추적(Ctrl+C로 종료) kubectl -n synapse-local logs -f deploy/platform-svc # 서비스 로그 추적 kubectl -n synapse-local describe pod -l app.kubernetes.io/name=platform-svc # 이벤트/에러 디버그

브라우저 GUI로 파드·로그·리소스를 클릭하며 볼 수 있습니다.

minikube dashboard # 브라우저 자동 오픈 minikube dashboard --url # URL만 출력(원격/수동 오픈용)

3.6 k8s에서 서비스 사용·데이터·이벤트 확인

compose 사용법(§6)의 k8s 버전입니다 — docker exec 대신 kubectl exec, 접근은 port-forward를 씁니다.

kubectl -n synapse-local port-forward svc/gateway 8080:80 # 한 터미널 유지 curl http://localhost:8080/api/platform/actuator/health # → platform-svc curl http://localhost:8080/api/knowledge/actuator/health # → knowledge-svc # learning-ai Swagger: 별도 터미널에서 kubectl -n synapse-local port-forward svc/learning-ai 8000:80 # 그 뒤 브라우저 http://localhost:8000/docs
kubectl -n synapse-local exec -it deploy/postgres -- psql -U synapse -d synapse -c "\dt" kubectl -n synapse-local exec -it deploy/redis -- redis-cli -a redis_local keys '*' kubectl -n synapse-local exec -it deploy/opensearch -- curl -s localhost:9200/_cat/indices?v
kubectl -n synapse-local exec -it deploy/kafka -- kafka-console-consumer --bootstrap-server localhost:9092 --topic knowledge.note.note-created-v1 --from-beginning

한 터미널에서 위 구독을 켜 두고 다른 터미널에서 액션(API 호출)을 일으키면 이벤트가 실시간으로 찍힙니다 — MSA가 파드 간 이벤트로 도는 걸 눈으로 확인할 수 있습니다.

정적 렌더만 먼저 확인하려면 (심화)
클러스터 없이 매니페스트가 올바른지 보려면 kubectl kustomize local-k8s. 인프라 5개 + Gateway + 앱 5개 Deployment(총 11)가 synapse-local 네임스페이스로 렌더되고 ExternalSecret/ghcr/AWS 호스트가 없어야 정상입니다. 실 API 검증은 kubectl apply -k local-k8s --dry-run=server.
트러블슈팅 (심화)
JVM 서비스가 반복 재시작(CrashLoop/Restarts 증가): 대개 메모리 부족입니다. opensearch+kafka+JVM 4개가 동시에 떠서 4GB로는 liveness가 파드를 죽입니다 → minikube delete && minikube start --memory=8192 --cpus=4 후 재기동(스크립트 기본값도 8GB). ImagePullBackOff: 이미지가 minikube에 적재 안 됨 → minikube image ls | grep synapse 확인, 없으면 minikube image load synapse-<name>:local 재실행. 토픽 미생성: kubectl -n synapse-local logs job/kafka-topics-init. Gateway 502/503: 대상 서비스가 아직 Ready 아님 → kubectl -n synapse-local get pods 확인. compose와 동시 실행 금지: 포트 충돌.
learning-ai / learning-card 이미지: 두 Dockerfile은 synapse-learning-svc의 dev 브랜치에서 수정 완료(learning-ai는 app/ 복사 순서, learning-card는 멀티스테이지 Dockerfile 신설). 빌드하려면 해당 레포를 dev로 두거나 main에 반영하세요. 미반영 상태면 두 파드만 ImagePullBackOff이고 나머지는 정상(스크립트는 한 이미지 실패해도 계속 진행). 검증 결과 수정된 Dockerfile + 8GB 메모리에서 11개 파드 전부 Running 확인.
compose 경로와 무엇이 다른가? (심화)
compose 경로(§4/§5)는 인프라를 호스트 컨테이너로 띄우지만, 이 경로는 인프라·Gateway까지 클러스터 안에 띄워 파드가 서비스 DNS(postgres,kafka,platform-svc 등)로 통신합니다. 실 배포(EKS)와 가장 가깝습니다.

4(대안) 풀 컨테이너 — docker compose

① synapse-gitops/docker-compose.yml로 인프라 + 5개 서비스를 한 번에 띄웁니다. 서비스는 형제 레포 소스에서 빌드됩니다. “일단 전부 떠 있는 모습”을 빠르게 보고 싶을 때 좋습니다.

로컬 기본값으로 동작합니다(외부 API 키는 mock 기본값).

cd C:\workspace\team-project-final\synapse-gitops cp .env.example .env

최초 실행은 모든 서비스 이미지를 빌드하므로 수 분 걸릴 수 있습니다(이후엔 빠름).

docker compose up -d --build
docker compose ps
curl http://localhost:8080/actuator/health # platform-svc curl http://localhost:8082/actuator/health # engagement-svc curl http://localhost:8083/actuator/health # knowledge-svc curl http://localhost:8084/actuator/health # learning-card curl http://localhost:8000/health # learning-ai (FastAPI) curl http://localhost:8085/subjects # schema-registry
docker compose down # 데이터 유지 docker compose down -v # 볼륨까지 삭제 (완전 초기화)
--build는 무슨 일을 하나? ghcr 이미지와의 관계 (심화)
각 서비스는 compose에서 build.context: ../synapse-*로 정의되어 있어 --build 시 형제 레포의 Dockerfile로 로컬 빌드합니다. build를 생략하면 image:에 적힌 ghcr.io/team-project-final/... 이미지를 받으려 시도합니다(레지스트리 접근/발행 필요). 신규 팀원은 로컬 빌드(--build)가 가장 확실합니다.
메모리가 부족하면 (심화)
인프라 6개 + 앱 5개를 동시에 띄우면 메모리를 꽤 씁니다(권장 8GB+). Docker Desktop의 Settings → Resources에서 메모리를 올리세요. 검색엔진은 JVM 힙이 커서 가장 먼저 영향을 받습니다(→ §8).
Gateway가 없는데 어떻게 호출하나? (심화)
경로①의 compose에는 Gateway가 없고 각 앱 포트(8080/8082/8083/8084/8000)가 호스트에 직접 노출됩니다. 그래서 위 헬스 체크도 서비스 포트로 바로 호출합니다. Gateway 경유 라우팅을 보고 싶다면 경로②(하이브리드)에서 Gateway(:8080)를 띄우세요.

5(대안) 하이브리드 — compose 인프라 + 소스 실행

② 인프라는 컨테이너로, 내가 수정하는 서비스만 소스로 직접 실행합니다. 코드 수정 → 재기동 루프가 빨라 실제 개발에 적합합니다. synapse-shared/docker-compose.yml을 씁니다.

cd C:\workspace\team-project-final\synapse-shared docker compose up -d postgres redis zookeeper kafka schema-registry opensearch kafka-init
docker compose ps docker exec -it synapse-kafka kafka-topics --list --bootstrap-server localhost:9092 curl http://localhost:8086/subjects # schema-registry curl http://localhost:9200/_cluster/health # opensearch

kafka-init가 토픽 5개(platform.auth.user-registered-v1, knowledge.note.note-created-v1, knowledge.note.note-updated-v1, learning.card.review-completed-v1, learning.ai.cards-generated-v1)를 자동 생성합니다.

내가 개발할 서비스 레포에서 Gradle 래퍼로 실행합니다(예: platform-svc). engagement / knowledge / learning-card도 같은 방식입니다.

cd C:\workspace\team-project-final\synapse-platform-svc ./gradlew bootRun

Python 3.12+가 필요합니다. 가상환경을 만들고 pyproject.toml 기준으로 설치합니다.

cd C:\workspace\team-project-final\synapse-learning-svc\learning-ai python -m venv .venv .venv\Scripts\activate # macOS/Linux: source .venv/bin/activate pip install -e . uvicorn app.main:app --reload --port 8090
curl http://localhost:8080/actuator/health # Spring 서비스(예: platform-svc) curl http://localhost:8090/health # learning-ai
왜 compose의 앱이 stub인가? (심화)
synapse-shared compose의 앱 서비스(platform/engagement/knowledge/learning-card/learning-ai)는 sleep infinity 스텁입니다. 실제 앱은 IDE나 ./gradlew bootRun으로 직접 실행해 코드 수정 → 재기동 루프를 빠르게 돌리는 것이 이 경로의 목적이기 때문입니다. 인프라(DB·Kafka·검색)만 컨테이너로 안정적으로 띄워두고, 앱은 본인이 잡습니다.
포트가 충돌해요 (심화)
synapse-shared compose의 Gateway가 :8080을 쓰는데 platform-svc 기본 포트도 8080이라 전체를 다 올리면 충돌할 수 있습니다. 그래서 1단계에서 인프라만 올립니다. 서비스를 다른 포트로 띄우려면 ./gradlew bootRun --args='--server.port=8181'처럼 지정하세요.
synapse-shared를 mavenLocal에 publish해야 하나? (심화)
아니요. platform-svc·knowledge-svc·gateway의 build.gradle.kts는 mavenCentral()만 쓰고 com.synapse:shared Maven 의존성이 없습니다. 즉 서비스는 단독으로 컴파일됩니다. synapse-shared의 역할은 (1) 로컬 인프라 compose, (2) Avro 스키마 정의, (3) Schema Registry 등록 스크립트 제공입니다. “shared를 먼저 빌드해야 컴파일된다”는 말은 이 프로젝트엔 해당하지 않습니다.
Kafka 토픽은 누가 만드나? (심화)
kafka-init 컨테이너가 기동 시 위 5개 토픽을 --if-not-exists로 자동 생성하고 종료합니다. 수동 생성이나 Avro 스키마 등록은 synapse-shared/scripts/의 create-kafka-topics.sh, register-schema.ps1을 참고하세요.

6서비스 사용·확인하기

설치(§3 k8s 또는 compose §4/§5)로 서비스가 떴다면, 이제 실제로 열어보고 호출하고 이벤트가 흐르는 걸 확인해 봅시다. k8s·하이브리드는 Gateway(:8080)를 경유하고, 풀 컨테이너(compose)는 서비스 포트를 직접 호출합니다.

5.1 브라우저로 열어보는 뷰

⭐ learning-ai Swagger UI — 이 프로젝트에서 버튼으로 API를 직접 실행("Try it out")해볼 수 있는 유일한 GUI입니다.

http://localhost:8000/docs # 경로① Swagger UI http://localhost:8090/docs # 경로② Swagger UI http://localhost:8000/redoc # ReDoc (읽기용 문서)

브라우저로 바로 열리는 상태(JSON) 뷰:

http://localhost:8080/actuator/health # platform-svc 상태 http://localhost:8085/subjects # 등록된 Avro 스키마 (경로② :8086) http://localhost:9200/_cluster/health # 검색엔진 상태
다른 서비스는 왜 Swagger UI가 없나? (심화)
Spring 서비스(platform·engagement·knowledge·learning-card)에는 springdoc이 적용돼 있지 않아 Swagger UI가 없습니다. 대신 /actuator/health 등 actuator 엔드포인트로 상태를 확인합니다. learning-ai만 FastAPI 기본 제공으로 /docs가 열립니다.

5.2 API 호출해보기 — Gateway 경유 vs 직접

경로②는 Gateway가 /api/{서비스}/** 요청을 받아 해당 서비스로 전달합니다(앞 2개 세그먼트 제거 후 전달).

curl http://localhost:8080/api/platform/actuator/health # → platform-svc curl http://localhost:8080/api/knowledge/actuator/health # → knowledge-svc

경로①은 Gateway가 없으니 서비스 포트로 직접 호출합니다.

curl http://localhost:8080/actuator/health # platform-svc 직접 curl http://localhost:8083/actuator/health # knowledge-svc 직접
Gateway 라우팅 규칙 (심화)
Gateway(synapse-gateway)는 4개 경로를 stripPrefix(2)로 전달합니다: /api/platform/**→platform-svc, /api/engagement/**→engagement-svc, /api/knowledge/**→knowledge-svc, /api/learning/**→learning-card. 예: /api/platform/actuator/health → platform-svc의 /actuator/health. (Redis 기반 rate limit 적용)

5.3 Kafka 이벤트가 흐르는 걸 보기

한 터미널에서 토픽을 구독해 두고, 다른 터미널에서 액션을 일으키면 이벤트가 실시간으로 출력됩니다 — MSA가 이벤트로 도는 걸 눈으로 확인할 수 있습니다.

docker exec -it synapse-kafka kafka-console-consumer --bootstrap-server localhost:9092 --topic knowledge.note.note-created-v1 --from-beginning

자동 생성되는 토픽 5개:

platform.auth.user-registered-v1 knowledge.note.note-created-v1 knowledge.note.note-updated-v1 learning.card.review-completed-v1 learning.ai.cards-generated-v1

5.4 저장된 데이터 직접 확인

docker exec -it synapse-postgres psql -U synapse -d synapse -c "\dt"
docker exec -it synapse-redis redis-cli -a redis_local_pw keys '*' # 경로② (경로①: -a redis_local)
curl http://localhost:9200/_cat/indices?v
자격증명은 어디서 오나? (심화)
두 스택 모두 PostgreSQL 사용자/DB는 synapse/synapse이며 컨테이너 내부 로컬 접속은 비밀번호가 필요 없습니다. Redis 비밀번호만 다릅니다 — 경로②(synapse-shared)는 redis_local_pw, 경로①(synapse-gitops)는 redis_local. 값은 각 docker-compose.yml·.env 기본값입니다.
